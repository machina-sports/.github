# Deploy flow — `dev → staging → production`

Canonical lifecycle for `machina-sports/*` apps deploying to AKS.

## Environments

| Env | Trigger | Cluster | Approval |
|---|---|---|---|
| `dev` | `push` to `development` branch | local / per-app | none |
| `staging` | tag `v.staging-<scope>.<N>` | `mks-community-cluster` | none |
| `production` | tag `v.production-<scope>.<N>` | `mks-community-cluster` | manual (GitHub Environment) |

`<scope>` is typically `main` (e.g. `v.staging-main.42`). Some repos use a feature scope (e.g. `v.staging-feat-auth.3`) when deploying a branch to staging without merging to `main`.

## Branching

- Feature work: `feat/<topic>`, `fix/<topic>`, `chore/<topic>`
- PR → `staging` (default integration branch)
- After staging soaks: `staging` → `main` via PR (often labeled `merge/staging-to-main`)
- Hotfix: `hotfix/<topic>` → `main` via PR + cherry-pick to `staging`

See [machina-sports/docs CONTRIBUTING.md](https://github.com/machina-sports/docs/blob/main/.github/CONTRIBUTING.md).

## Tag conventions

```
v.staging-main.42        # main-line staging build #42
v.staging-feat-auth.3    # ad-hoc staging from feature branch
v.production-main.150    # production deploy
```

Generated by `semantic-release` (configured per repo) on push to `staging` / `main`.

## Lifecycle

```
                ┌─ dev ─┐         ┌─ staging ─┐         ┌─ production ─┐
push develop ──▶│ local │         │           │         │              │
                └───────┘         │           │         │              │
                                  ▲           │         ▲              │
                       tag v.staging-*        │         │              │
                                              │  tag v.production-*    │
                                              │         │              │
   reusable-build-staging.yml ────────────────┘         │              │
       (builds + deploys to AKS)                        │              │
                                                        │              │
   reusable-release-production.yml (gated) ─────────────┘              │
       (manual approval → AKS prod)                                    │
```

## Promoting a build

1. Merge feature PRs into `staging`.
2. `semantic-release` (CI) tags `v.staging-main.<N+1>` automatically.
3. `reusable-build-staging.yml` builds image, deploys to AKS staging.
4. Soak in staging. Smoke-test.
5. PR `staging` → `main`. Merge.
6. On `main`: `semantic-release` tags `v.production-main.<N>`.
7. `reusable-release-production.yml` triggers, **waits for manual approval** in the `production` environment.
8. Reviewer approves → AKS production deploy.

## Per-app overrides

Each app's `kubectl set image` parameters (namespace, deployment-name, container-name) live in the **caller workflow**, not in the reusable. Examples:

| App | Namespace | Deployment | Container |
|---|---|---|---|
| `machina-studio` | `machina-workspace` | `machina-workspace-client-services-{staging,production}` | `machina-workspace-studio-{staging,production}` |
| `machina-core-api` | `machina-workspace` | (see core-api `.github/workflows/`) | |
| `sportingbot-web` | `tenant-entain-organization` | (see sbot `.github/workflows/`) | |

## Rollback

Re-run `reusable-release-{staging,production}.yml` via `workflow_dispatch` with the prior known-good `image-tag`. AKS rolls forward to the requested image; effectively a rollback when the prior tag is older.

For studio, the existing `/studio:update-staging` and `/studio:update-production` Claude Code skills wrap this same operation with extra safety checks (image existence, double confirmation for prod). They remain valid alongside the reusable workflow.
